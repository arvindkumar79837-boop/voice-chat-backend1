## Problem

`firebaseAuth.controller.js` had 4 instances of `.replace('+91', '')` that stripped India's country code before storing phone numbers. This broke international numbers (+971, +1, +44, etc.) — they'd be stored with their country code while Indian numbers were stored without, causing inconsistent data and failed lookups.

---

## What Changed

### Backend (`repo2`)

1. **`firebaseAuth.controller.js` — Removed all 4 `.replace('+91', '')` calls:**
   - Line 66: User lookup `{ phone: firebasePhone?.replace('+91', '') }` → `{ phone: firebasePhone }`
   - Line 84: User creation `phone: firebasePhone?.replace('+91', '')` → `phone: firebasePhone`
   - Line 106: Existing user update `user.phone = firebasePhone.replace('+91', '')` → `user.phone = firebasePhone`
   - Line 235: Link Firebase account `user.phone = decodedToken.phone_number.replace('+91', '')` → `user.phone = decodedToken.phone_number`
   - All phone numbers now stored as full E.164 (e.g. `+919876543210`, `+971501234567`, `+14155552671`)

2. **`src/utils/phoneUtils.js` — NEW utility file:**
   - `formatPhoneForDisplay(phone, options)` — Formats E.164 for display (strips country code, adds spacing)
   - `maskPhone(phone)` — Masks phone for privacy, handles E.164 properly (e.g. `+91****543210`)
   - `formatLocalNumber(local, mask)` — Formats 10-digit local number with spacing

3. **`profileController.js` — Updated `maskPhone` helper:**
   - Replaced inline implementation with import from `phoneUtils.js`
   - Now handles E.164 format globally (Indian, US, UAE, etc.)

## Data Format

| Before | After |
|--------|-------|
| `9876543210` (India only) | `+919876543210` (India) |
| `+971501234567` (broken) | `+971501234567` (UAE) |
| `+14155552671` (broken) | `+14155552671` (USA) |

**Rule:** Phone numbers are ALWAYS stored in full E.164 format (`+<countrycode><number>`). Display formatting is done only in UI, never in stored data.

## Existing User Migration

Existing users with Indian numbers stored as `9876543210` (without `+91`) will continue to work for login (Firebase returns full E.164, but the phone field lookup is exact-match). New users and re-logins will store full E.164.

To migrate existing data (optional):
```js
db.users.updateMany(
  { phone: { $regex: /^[0-9]{10}$/ } },
  [{ $set: { phone: { $concat: ["+91", "$phone"] } } }]
)
```
